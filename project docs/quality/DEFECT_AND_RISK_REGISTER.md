## CRITICAL Findings

### FR-001: No Automated Tenant Isolation at ORM Level

**Severity:** CRITICAL
**Confidence:** HIGH
**Affected:** All apps
**Evidence:** `TenantScopedManager.for_tenant()` is opt-in, not default. `TenantAwareModel.tenant_id` is a UUIDField, not a ForeignKey. No middleware injects tenant_id.
**Runtime impact:** A single forgotten `tenant_id` parameter in a new service creates a cross-tenant data leak.
**Why it matters:** Multi-tenant isolation is the foundational security guarantee. Currently enforced only by developer discipline.
**Suggested action:** Consider middleware-based tenant injection or row-level security (PostgreSQL RLS).

### FR-002: RBAC Enforcement Can Be Disabled Per-Tenant

**Severity:** CRITICAL
**Confidence:** HIGH
**Affected:** All permission-gated operations
**Evidence:** `apps/kernel/services/permission_service.py:135` — if `RBACConfiguration.get_enforcement_enabled()` is False, `require()` returns immediately.
**Runtime impact:** Setting `rbac.enforcement.enabled=false` in ConfigurationValue table disables ALL RBAC for that tenant.
**Why it matters:** No audit alert when enforcement is toggled. A compromised config change silently bypasses all permissions.
**Suggested action:** Add audit logging when enforcement is toggled. Consider removing the toggle in production.

### FR-015: Public Caregiver and Organization Directory Views Failed to Honor Explicit Tenant Hints — **RESOLVED 2026-07-18**

**Severity:** HIGH
**Confidence:** HIGH (both symptoms reproduced against a real running server, root-caused via direct code inspection, each fixed with a failing-then-passing regression test)
**Affected:** `apps/public_site/views.py` (`find_a_caregiver`, `find_an_organization`)
**Evidence:** Both public *detail* views (`caregiver_profile()`, `organization_profile()`) resolve an explicit `?tenant=<slug>` query hint via the single shared `_resolve_optional_tenant_hint()` helper and pass the resolved `tenant_id` into their respective profile services — added specifically so a supplier living in a non-default tenant (e.g. `seed_product_walkthrough`'s own dedicated demo tenant) can be previewed. Neither *directory/listing* view (`find_a_caregiver()`, `find_an_organization()`) adopted the same hint resolution: both called their directory service (`CaregiverDirectoryService.search()` / `OrganizationDirectoryService.search()`) without a `tenant_id` argument at all, so each service silently defaulted to `TenantService.get_default_tenant_id()` regardless of the query string. An entity in any non-default tenant — proven ACTIVE, VERIFIED, ACTIVE `ServiceSupplier`, correct type, publicly visible per `common.is_publicly_visible_attrs()`, confirmed 200 + fully rendered at its own direct detail URL — was therefore invisible in the corresponding directory for that same tenant.

**Caregiver symptom (found and fixed first):** `GET /find-a-caregiver/?tenant=demo-senior-platform` rendered `0 مراقب یافت شد` with no filters applied, while `GET /find-a-caregiver/<id>/?tenant=demo-senior-platform` rendered the same caregiver correctly.

**Organization symptom (latent — masked by an unrelated, separate seed-data limitation):** the same defect exists in `find_an_organization()`, but was invisible in ordinary manual testing because every organization `seed_product_walkthrough` creates has `verification_status=unverified`, which already excludes them from the directory for an unrelated reason (`common.is_publicly_visible_attrs()`'s verification predicate). Proven directly, not assumed: a real ACTIVE, VERIFIED, ACTIVE-supplier organization was constructed in the demo tenant via the canonical `get_or_create_supplier_for_organization()` bridge — `GET /find-an-organization/?tenant=demo-senior-platform` rendered `0 سازمان یافت` for it, while `GET /find-an-organization/<id>/?tenant=demo-senior-platform` rendered it correctly (200, fully rendered) — the identical directory-vs-detail contrast already proven for caregivers. The verification-status limitation is real and remains unfixed (see Remaining Known Risks below) but is a **distinct, unrelated cause** from the tenant-hint gap — fixing the tenant-hint gap does not and must not make unverified organizations publicly visible, and it does not: the constructed test organization was explicitly VERIFIED to isolate and prove the tenant-hint defect specifically.

**Shared root cause:** the tenant-hint resolution helper was wired into both detail views when the multi-tenant preview feature was introduced, but was never symmetrically wired into either directory view — an omission, not a design decision (both directory services already correctly accept and use a `tenant_id` keyword argument, confirmed by their own existing service-level tests, which pass `tenant_id` explicitly; the gap was entirely in the view layer, not the service or query layer). Root-cause proven stage-by-stage for both entity types: querying with `tenant_id=TenantService.get_default_tenant_id()` excluded the known entity at the very first stage (initial candidate population); querying with the correct tenant included it through every subsequent stage.

**Runtime impact:** An entity correctly activated, verified, and supplier-synchronized by the Core Profile-ServiceSupplier Invariant Remediation (FR-014/PR #18) was still invisible in the one surface most visitors actually use — the directory listing — even though its own direct profile URL worked. For organizations specifically: once an organization becomes verified through the normal verification workflow, it would have silently reproduced the same class of failure the caregiver fix already resolved, had this not been proactively fixed here rather than left masked behind the current seed-data limitation.

**Resolution:** `find_a_caregiver()` and `find_an_organization()` both now call `_resolve_optional_tenant_hint(request)` — the same helper both detail views already use — and pass the resolved `tenant_id` into their respective directory service. A two-line change per view, symmetrical, no duplicated tenant-resolution logic (a single shared helper, already defined once in this module, now used identically by all four public views). No service, query, or visibility-rule change in either case; no organization seed-verification behavior changed; no unverified organization was made publicly visible. Callers that never pass `?tenant=` get exactly the pre-existing default-tenant behavior for both entity types; an unknown slug now correctly 404s on both directories instead of silently rendering an empty list, matching both detail views' own established contract.

Regression tests, symmetrical for both entity types (`FindACaregiverViewTenantHintTest` + `FindAnOrganizationViewTenantHintTest`, 8 tests each, mirroring `CaregiverProfileTenantHintTest`/`OrganizationProfileTenantHintTest`): correct hint lists the entity; missing hint still excludes a non-default-tenant entity; unknown slug 404s; wrong tenant hint cannot cross-tenant-leak; unverified/DRAFT/inactive-supplier entities remain excluded even with a correct hint; and a consistency test proving an entity visible via the detail-page hint is also visible via the directory hint for the same unfiltered request. All 16 confirmed failing before their respective fix and passing after. Full regression 2321 -> 2329/2329 green (caregiver fix) -> **2337/2337 green** (organization fix), 0 failures, 0 errors, no warnings at each step. Branch `fix/public-caregiver-directory-visibility`, PR #19 **merged to `main`** (merge commit `aac018575a2f9380d75a11c55bf08b34ce65a511`, 2026-07-18).

### FR-016: Public Directory Views Fixed Tenant-Hint *Resolution* (FR-015), but Directory-Generated Navigation Still Dropped It — **RESOLVED 2026-07-19**

**Severity:** HIGH
**Confidence:** HIGH (reproduced against a real running server, root-caused via direct code inspection, fixed with 6 failing-then-passing regression tests covering cards/pagination/filters/reset for both entity types)
**Affected:** `apps/public_site/services/directory_service.py`, `apps/public_site/services/organization_directory_service.py`, `apps/public_site/services/common.py`, `apps/public_site/services/viewmodels.py`, `apps/public_site/views.py`, `templates/public_site/caregiver_directory.html`, `templates/public_site/organization_directory.html`
**Distinction from FR-015:** FR-015 (PR #19) fixed tenant *resolution* at the two directory **views**' own boundary — `find_a_caregiver()`/`find_an_organization()` now correctly read `?tenant=<slug>` and scope their own query to it. This follow-up fixes tenant-*context propagation* through the **URLs those directories generate** for further navigation (caregiver/organization card links, pagination links, the filter form, the "clear filters" link) — a distinct defect at the template/URL-building layer, not the tenant-resolution layer FR-015 already closed.
**Evidence:** `GET /find-a-caregiver/?tenant=demo-senior-platform` correctly renders 200 with real caregivers (FR-015's fix working as intended). But every rendered caregiver card link (`CaregiverCardViewModel.profile_url`, built in `CaregiverDirectoryService._build_card()`) was a hardcoded `f"/find-a-caregiver/{supplier.id}/"` with no query string at all — clicking any card took the visitor to `GET /find-a-caregiver/<supplier-id>/` (no hint), which resolves the *default* tenant and 404s for a supplier that lives anywhere else. The same gap existed in pagination links (`_build_pagination()`'s `params` dict never included `tenant`), the filter `<form method="get">` (no hidden `tenant` field, so a GET submission drops the query string entirely, keeping only the form's own named inputs), and the "clear filters" empty-state link (hardcoded `action_href="/find-a-caregiver/"`). Identical shape confirmed in `find_an_organization()`/`OrganizationDirectoryService`.
**Runtime impact:** A visitor who reached the directory via a valid, explicit tenant hint (the only way to browse a non-default tenant at all, e.g. a demo/preview link) could see results but could not click into any of them, use pagination, resubmit a filter, or clear filters without the entire browsing session silently falling back to the wrong (default) tenant — a broken, not merely inconvenient, navigation experience for every non-default-tenant visitor.
**Resolution:** Added one shared helper, `common.append_tenant_query(url, tenant_slug)` — returns `url` unchanged when no hint is active, otherwise appends `?tenant=<slug>`; navigation context only, never trusted as authoritative (the destination view still independently re-resolves and re-validates the hint via the same `_resolve_optional_tenant_hint()` FR-015 already established). Both directory services now accept an optional `tenant_slug` parameter (the view passes the already-validated raw slug string alongside the already-resolved `tenant_id` — the slug is never used for query filtering, only for URL construction) and thread it through: `_build_card()` appends it to `profile_url` via the shared helper; `_build_pagination()`'s `params` dict includes `tenant` when present (reusing the pre-existing, unmodified `common.build_pagination()` query-string-preservation mechanism — no new pagination-URL logic); `_build_filters()` exposes the slug as a new `tenant_slug` field plus a computed `reset_url` field on `DirectoryFiltersViewModel`/`OrganizationDirectoryFiltersViewModel`, so both directory templates render a hidden `<input type="hidden" name="tenant" value="...">` inside the filter form (a GET-form hidden field is resubmitted automatically with every filter change) and use `reset_url` for the "clear filters" link instead of a hardcoded path. Zero behavior change for `featured()` (Home Page) and `build_cards_for_supplier_ids()` (Customer Favorites) — neither passes `tenant_slug`, so their card URLs are byte-identical to before.
**Tests:** 8 new (`DirectoryTenantLinkPropagationTest`): caregiver card links carry the hint and resolve (200); caregiver pagination links carry the hint and resolve; caregiver filter form's hidden field carries the hint and a resubmission stays scoped to the same tenant; caregiver "clear filters" link carries the hint and resolves; missing/wrong tenant context still excludes cross-tenant caregivers (no leak); the organization-side equivalents (card links + combined pagination/filter/reset); unverified organizations remain absent regardless of the hint. 6 of 8 confirmed failing against the pre-fix code (verified by temporarily stashing the production diff and running the new tests against the original, unmodified views/services) and all 8 passing after; the other 2 (leak-prevention, unverified-exclusion) correctly passed both before and after, since they assert the absence of something the fix was never meant to change.
**Verification:** `git diff --check` / `manage.py check` / `manage.py migrate --check` all clean. Focused suites (`apps.public_site`, `apps.discovery`, `apps.kernel.tests.test_supplier_registry`, `apps.kernel.tests.test_supplier`, `apps.kernel.tests.test_seed_product_walkthrough`, `apps.accounts.tests.test_profile_supplier_invariant`) — 374/374 pass. Full regression: **2337 -> 2345/2345 green**, 0 failures, 0 errors, no warnings. Manual runtime verification against a real local server (re-seeded via `seed_product_walkthrough`, plus a disposable verified organization created and deleted for the organization-side check, never committed): 3 caregiver card links followed successfully (200 each); search/city/service/cooperation filters all preserved the hint via the hidden field; "clear filters" preserved the hint; an invalid tenant slug still 404s; the organization card link, filter hidden field, and follow-through all confirmed working identically.
**Preserved, unchanged:** tenant isolation (the destination view is always the authority — a manipulated or stale `tenant` query parameter is re-validated on every request, never trusted because it arrived via a link or hidden field), the default-tenant fallback for hint-less requests, unknown-slug 404 behavior, and every public-visibility predicate (ACTIVE profile, VERIFIED, ACTIVE supplier, correct type, active membership for affiliated caregivers) — none of `CaregiverDirectoryService._filter_candidates()`/`OrganizationDirectoryService._filter_candidates()`/`common.is_publicly_visible_attrs()` was touched.
