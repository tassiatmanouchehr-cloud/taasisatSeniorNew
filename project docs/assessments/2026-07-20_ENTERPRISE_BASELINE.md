## 1. Executive summary

The platform is substantially real: registration, verification, activation, RBAC,
the caregiver/organization public marketplace, the order/booking/execution/review
lifecycle, and the design system are all implemented against genuine services,
tested in depth, and largely match their documentation. The gaps cluster in three
places — **money leaving the system** (no real PSP, commission math never wired to
settlement, escrow release has no consumer), **getting users in** (OTP has no SMS
provider), and **operating the system** (no log/config/job visibility in the admin
portal, no production deployment infrastructure, CI never executed).

Documentation quality is unusually high for this codebase — of roughly 40 specific
claims independently checked against code across all eight investigations, the
large majority matched exactly, including precise test counts, exact resolution
orders, and exact query-count figures. The handful of confirmed mismatches are
concentrated in one file (`PORTALS_AND_APIS.md`'s stale Admin Portal route count)
and one systemic omission (the Financial Engine's own documentation describes the
commission and escrow-release machinery as if it runs money through it, without
disclosing that the calculator and the release consumer are both unwired). See
§10 for the full list.

## 3. Domain findings (detail)

### 3.1 Identity & Access

Phone/OTP-only registration and login, a real multi-stage document-verification and
activation pipeline, and an RBAC system with unusually strong self-enforcing
guardrails (AST inspection, not just docstrings). The one hard production blocker
in this domain is that OTP codes are never actually sent anywhere.

- **Registration** — `apps/accounts/services/registration.py`: `RegistrationService.create_customer/create_caregiver/create_company_admin`, all atomic; new profiles start `DRAFT`. Optional company affiliation at caregiver registration. Idempotent multi-role attach via `ensure_customer_profile()`/`ensure_caregiver_profile()`. 51 tests. No CAPTCHA anywhere; phone/OTP only, no email path.
- **Login** — `login_view()` → OTP request/verify → `auth_login()` with explicit `ModelBackend`. Separate email/password path for Django-admin staff. 33 tests. No brute-force/rate-limit library beyond OTP's own 5/hour + 90s cooldown.
- **OTP** — `OTPChallenge` (SHA-256 hash, expiry, attempts, IP/UA capture), `OTPService.request_otp()/verify_otp()`, 7 tests. DEBUG-only dev-code echo. **`_send_sms()` is referenced only in a comment naming Kavenegar as the intended provider — the method is never defined anywhere in the codebase.** Outside `DEBUG=True`, a challenge is generated and stored but delivered to no one. Risk: **High** — production blocker for the entire phone-based identity system.
- **Verification workflow** — `VerificationDocument` (exactly-one-owner DB `CheckConstraint`, 9 document types), `DocumentService.upload_*()/resubmit()` (content-sniffs files, never trusts client content-type), `VerificationReviewService.approve()/reject()/request_correction()` (row-locked, self-review blocked, tenant re-derived from owner before permission check), `RequiredDocumentPolicy`, `ProfileVerificationRollupService`. 83 tests. Missing: customer document verification does not exist as a domain concept; no per-service-category document requirements; no automated/OCR authenticity checking.
- **Approval / activation** — `ActivationEligibilityService` (read-only, 100% completion + VERIFIED required), `ProfileActivationService` (row-locked, real `DRAFT→ACTIVE` transition, syncs `ServiceSupplier` in the same transaction, idempotent repair on re-activation). 50 tests. Missing: no suspension/deactivation/reactivation workflow (deferred, BG-019); no periodic re-verification trigger. Risk: **Medium** — FR-014 documents this invariant was fixed twice and needed a strengthened AST guardrail plus two standalone reconciliation commands, evidence of past fragility even though currently resolved.
- **Roles** — `Role`/`RoleAssignment` with a DB `UniqueConstraint` against duplicate active grants. Two parallel catalogs: `DEFAULT_TENANT_ROLES` (12, underscored slugs, seeded by `seed_auth_roles`) and `DEV_BOOTSTRAP_ROLES` (14, hyphenated slugs, seeded by `seed_tenant`) — deliberately not merged. No UI for a tenant admin to create custom roles at runtime. Risk: **Medium** — acknowledged unresolved tech debt, and the source of a genuine documentation error (see §10).
- **Permissions (RBAC)** — 38 permission keys registered through one canonical registry. `PermissionService.check()/require()` fail-closed by default, documents its own two deliberate exceptions (`system_context`, `ownership_authorized_by`). A guardrail suite (10 tests) proves via AST/regex inspection that no raw permission-key string literal exists anywhere in production code. Risk: **Critical** — RBAC enforcement can be fully disabled per-tenant via `RBACConfiguration.get_enforcement_enabled()`, and there is no admin UI or dedicated audit alert covering changes to that flag itself. Risk: **High** — the `ownership_authorized_by` fallback trusts every caller completely by design; correctness depends entirely on every call site getting it right.
- **Authentication mechanics** — Standard Django session middleware; `Tenant` FK present on Person/UserAccount/Role/RoleAssignment; centralized default-tenant resolution. **No tenant-scoping or RBAC-enforcement middleware** — confirmed directly from the `MIDDLEWARE` list. Every view/service independently resolves `tenant_id`. Risk: **High** — isolation is a convention, not a structural guarantee; one forgotten `tenant_id` parameter is a cross-tenant leak, mitigated in practice by `TenantAwareModel` and per-app isolation tests but with no middleware-layer backstop.
- **Authorization enforcement** — every one of ~20 admin views calls `require_admin_permission()` first line, verified directly. Consistent self-action refusal and 404-not-403 cross-tenant pattern, verified directly in service code.

### 3.2 Customer Portal

A genuinely complete order-placement and account experience wrapped around a
payments surface that is honest about being simulated. Wallet and Settings are the
thinnest sub-areas — not buggy, simply read-only summaries standing in for pages a
customer would expect to interact with.

- **Dashboard** — aggregates orders, upcoming visits, care recipients, read-only wallet balance, notifications, profile completion, pending actions. Bounded-query-count regression test. 5 tests.
- **Profile** — `profile_view`/`profile_edit_view`, phone/email intentionally non-editable (OTP identity). No avatar upload.
- **Favorites** — `Favorite` model + idempotent `FavoritesService`; adding happens from `public_site`, portal only lists/removes (intentional split). 33 tests. Risk: **Low** — `build_list_view()` materializes the entire favorites queryset in Python before slicing for pagination (documented, accepted — KL-023).
- **Orders** — 7-step session-backed wizard, list/detail/history, share-link create/revoke with a public token-only view. 22 tests.
- **Invoices / payments** — `CustomerPaymentsPresentationService` wraps `FinancialDocumentService`; pay/approve/dispute all present, 9 tests. **The only registered payment provider is `PaymentProvider.FAKE`. The authenticated customer can POST `outcome=SUCCEEDED`/`FAILED` directly for their own payment** — honestly disclosed in `RUNTIME_WORKFLOWS.md` as "MOCKED PSP," in tension with the same document calling Phase 4 "production-complete."
- **Notifications** — list + unread count (2 tests, thinnest file in the app). **No mark-as-read anywhere** — confirmed by grep, zero hits.
- **Wallet** — no `/portal/wallet/` route exists. Balance embedded read-only in dashboard/payments only. No dedicated page, no transaction history, no top-up/withdraw.
- **Settings** — `settings_view` is a read-only summary (phone/email/status) linking back to Profile edit. Zero interactive elements. No password/PIN, no notification preferences, no session management, no account deletion.
- **IDOR protection** — confirmed sound. `resolve_customer_profile()` resolves exclusively from `request.user`, never URL/body. Dedicated `test_access_control.py` proves anonymous-403/authenticated-200 across 6 routes and 404-not-leak on cross-customer care-recipient/order access.
- **KL-022 confirmed** — no flash-message framework. `django.contrib.messages` is installed and wired into the base template but has zero production callers anywhere in the app tree. Every failing POST action silently no-ops with a redirect.

### 3.6 Marketplace / Public Site / Discovery

21 public routes, a single canonical visibility policy reused everywhere, and a
4-tier tenant resolver that both governing documents describe exactly as the code
implements it. The one real, previously-undetected risk: the query-*count* N+1
fix (KL-012) never addressed the underlying candidate-*set size*, which still has
no database-level limit before ranking.

- **Directories & Filtering** — caregiver and organization directories mirror each other's architecture on non-overlapping supplier types. 40+15 tests. Risk: **Medium** — `SupplierSearchService.filter_suppliers()` materializes *every* matching ACTIVE supplier into Python before ranking and pagination slice it to 12; `SearchQuery.limit/offset` are computed but never read by this path. Query *count* is flat (the KL-012 fix); result-set *size* is not bounded. Undocumented.

- **Public pages (21 routes)** — route count matches documentation exactly. A real architecture-guardrail test forbids any ORM call inside `public_site/views.py`.
- **Marketplace / tenant visibility** — 4-tier `resolve_public_tenant()`: explicit hint → optional env setting (loud `ImproperlyConfigured` if misconfigured) → `DEBUG`-only dev tenant → platform default. Used uniformly by all 5 tenant-aware views. 39 tests. Risk: **Low** — tenant resolution runs twice per request (view + context processor); idempotent, just duplicated work.

### 3.7 Order Workflow

Everything downstream of "an order exists" — matching, booking, execution,
reviews — is real, atomic, and concurrency-tested. The upstream "offer
marketplace" is the clearest Not Started finding in the whole audit: a
fully-modeled, migrated, 40-test-covered `OrderOffer` table with *zero* service,
view, or business logic anywhere touching it.

- **Order lifecycle & State machine** — 7 states, full `OrderStatusHistory` audit trail. `status_machine.py` is documented as "the only code that may mutate `Order.status`" and verified true — every caller delegates rather than writing the field directly.
- **Matching & Booking** — `MatchOrchestrator` proposes, `AssignmentService` decides — matching never writes `assigned_supplier` directly. Booking has a dedicated `test_concurrency.py` + `test_atomicity.py`. 67 assignment tests, 33 matching tests.
- **Cancellation** — `request_cancellation()`/`approve_cancellation()` exist and transition state correctly, but **neither function contains a `PermissionService.require()` call**, unlike `AssignmentService.assign()` or `ExecutionService.close_session()` in the same domain. Risk: **High** — if no upstream caller enforces who may cancel, any actor able to reach these functions can force any non-final order to CANCELLED; orders/booking/matching/execution have zero Views of their own, so this depends entirely on portal/API callers doing the right thing — not independently verified.
- **Completion** — `ExecutionSession` deliberately avoids a plain "COMPLETED" state. `close_session()` is the sole path to `Order.COMPLETED` and does carry a real permission check — stronger authorization here than at the cancellation layer.
- **Reviews** — the one sub-area with a confirmed live HTTP surface (API + portal, not just services). 39 tests.
- **OrderOffer / Offer Marketplace** — `OrderOffer` model, migration, and DB constraints are real. `grep -rn "OrderOfferService"` across the entire repository returns **zero matches**. The 40-test file exercises only field defaults and constraint enforcement. Both `IMPLEMENTATION_ROADMAP.md` and `IMPLEMENTATION_STATE.md` state this plainly and correctly — the one Not Started item that is fully, accurately self-disclosed.

### 3.8 Financial Engine

The highest-risk domain, by nature and by finding. Individually, every
money-handling primitive is well-built — DB-level conservation constraints on
escrow, idempotency keys everywhere, append-only ledger entries. But three of
them terminate in mid-air: commission math is never applied, escrow release has
no consumer, and every payment is simulated. None of this is live-broken today
because every one of these paths is gated off by default — but the wiring
genuinely does not exist yet.

- **Invoices** — `FinancialDocument` enforces content-immutability once issued (DB-checked). No verify/approve concept — only issue/lock/cancel. 17 tests.
- **Payments** — idempotent intent/callback engine with real amount/currency validation and durable audit of every callback, accepted or rejected. **Exactly one provider adapter exists, and its own docstring says real PSPs "are deferred entirely."** 24 tests.
- **Settlement** — two disconnected mechanisms: (1) **Direct settlement** — the real, live path, triggered from every successful payment callback, posts ledger entries, credits the canonical wallet, retries via a background job on failure; (2) **`SettlementBatch`/`SettlementItem`** (net-position batching) — its own docstring says "no real payout/bank transfer happens here"; only `calculate_net_position()` and `create_batch()` exist, **no code anywhere transitions a batch to `APPROVED` or `PAID`**.
- **Escrow** — the best-engineered part of this domain: three `CheckConstraint`s enforce the conservation equation at the database level, all 6 methods atomic/locked/idempotency-keyed. Gated off by default. **Most significant finding in this assessment:** `ReleaseInstruction`'s own docstring states a future "PR-C" is the only allowed consumer to move it from `READY` to `CONSUMED`. A repository-wide grep for `ReleaseInstructionStatus.CONSUMED` and for "PR-C" implementation returns nothing — money held in production escrow can be marked releasable, but no code anywhere credits a beneficiary's wallet from a release. Currently latent (every tenant ships with this path off) but would surface immediately if enabled.
- **Wallet** — `apps.wallet` is the canonical, actively-used implementation. 34 tests. `apps.finance` also has a `WalletAccount`/`WalletTransaction` pair, explicitly labeled LEGACY/FROZEN in its own docstring with zero production callers. All three documentation sources describe this identically and correctly — the one area of full agreement across every source checked.
- **Commission** — full contract/dispute/objection/release/refund lifecycle, 132 tests, seeded default rate tiers. **`AllocationCalculator.allocate()` — the only code that computes a platform/company/caregiver split — has no callers anywhere except its own 9 unit tests.** `DisputeResolutionService.resolve()`, the one place money is actually divided, requires the caller to supply the three amounts explicitly rather than deriving them from the contract's configured rates.
- **Accounting (ledger)** — append-only, balanced-pair-enforced, deduped by a DB constraint. Only Direct Settlement posts entries — the escrow production path posts none at all.
- **Refunds** — `RefundInstructionService.initiate()` calls the Fake provider's simulated refund. Risk: **Medium** — the provider lookup is hardcoded to the literal string `"FAKE"` rather than the tenant's configured provider — invisible today because FAKE is the only option, a landmine for whoever wires in a real PSP later.
- **Reports** — real DB aggregation via `FinancialReportService.get_financial_summary()`, correctly reads the canonical `apps.wallet`. 37 tests. Narrow scope — one summary DTO only.

### 3.9 Infrastructure

Security, privacy, tenant isolation, and the design system are all in genuinely
good shape, each backed by real automated guardrail tests rather than convention
alone. Production operability is the gap: a CI pipeline that has (by the
project's own admission) never run, a dev-only Docker setup, and Django-level
production hardening with nothing to actually run it on.

- **Security** — zero `|safe` filter usages repo-wide; the only raw-SQL call is a parameterless health-check; CSRF middleware present with no `csrf_exempt` views found anywhere; secrets sourced from environment, no fallback in production.
- **Tenant isolation** — `TenantAwareModel` raises if saved without a tenant. Dedicated `test_tenant_isolation.py` files exist in 11 separate apps. An AST-based architecture-guardrail suite enforces sole-writer and ORM-discipline invariants as executed tests.
- **Notifications (providers)** — dispatch pipeline is real (locking, idempotency, job-queue integration). Only fake SMS/email/push providers exist — matches KL-002 exactly.
- **Background jobs** — `select_for_update(skip_locked=True)` claiming, exponential backoff, idempotency keys. Runs as a management command, not a scheduled Celery Beat task — needs external cron, none found in-repo.
- **Accessibility** — a real Playwright + axe-core WCAG 2.0/2.1 A/AA gate exists and is wired to fail CI. Runs against 14 design-system showcase pages only — not the live portal/public-site/booking templates, where aria-attribute density measured low (26 of 125 template files).
- **CI** — a genuinely comprehensive 5-job pipeline (lint, UI-quality, Tailwind build, Django tests, visual-regression) against real Postgres+Redis service containers. **The project's own defect register (FR-013) states it has never actually executed** — none of its gates have been proven to pass end-to-end outside local/manual runs.
- **Docker & Deployment** — a complete 5-service development compose exists. `config/settings/production.py` correctly hardens HSTS, SSL redirect, secure cookies, frame options; `requirements/prod.txt` lists gunicorn and sentry-sdk. **No production Dockerfile, no production compose variant, no nginx/reverse-proxy config, no CD pipeline, no infra-as-code, and no start script anywhere in the repository.** Matches KL-003 exactly.

## 5. Test coverage review

238 test files, on the order of 2,459 test methods repo-wide. Coverage is deep
and concurrency-aware in exactly the places that carry real risk (booking,
availability, affiliation, escrow) — and conspicuously thin or absent in exactly
the places flagged as gaps above.

**Well tested:** booking concurrency (dedicated `test_concurrency.py` +
`test_atomicity.py`, real `TransactionTestCase`s); availability overlap races (9
real concurrent-transaction tests); affiliation lifecycle (46 tests, DB-constraint-
backed); RBAC guardrails (AST/regex source inspection, not just behavior);
public-profile privacy (dedicated leak-prevention tests); tenant isolation
(dedicated test file present in 11 separate apps).

**Thin or absent:** `apps/common` (zero dedicated test files for the abstract
base classes every tenant-scoped model relies on); customer-portal notifications
(2 tests, thinnest file in that app); `OrderOffer` (40 tests, all model/
constraint-level — no service-layer behavior exists to test); `AllocationCalculator`
(9 unit tests prove the math in isolation; nothing tests it in a settlement/

permission check existing in that function); live-template accessibility (the
only automated gate covers the design-system showcase, not portal/booking flows).

**Highest-risk areas by test-coverage gap:** (1) escrow release consumption —
untestable today because the consumer doesn't exist; (2) commission allocation
in a live settlement path — well tested in isolation, zero coverage at the
integration point because there is no integration; (3) order cancellation
authorization — a real gap, since the function that can force any order terminal
has no permission-enforcement test because it has no permission enforcement.

## 6. Architecture review

No redesign is warranted — the layering is sound and, unusually, self-enforced
by tests rather than left to reviewer discipline.

- **Layer separation — holds.** Guardrail tests directly forbid ORM access inside `public_site`, `api`, and portal views; every mutation goes through a service. Enforced by source inspection, confirmed present and passing.
- **Domain boundaries — mostly holds, two known exceptions.** Wallet exists twice by deliberate, documented choice (one frozen). Settlement exists twice by what looks like historical accretion — a Module-05-era batch/netting primitive sits unused beside the real direct-settlement path with no reconciliation layer between them.
- **Tenant isolation — sound design, no structural backstop.** `TenantAwareModel` plus per-app isolation tests is strong and consistently applied, but entirely convention-enforced at the request layer — no middleware verifies a resolved tenant before a view runs.
- **Permission model — one deliberate trust boundary, used a lot.** The `ownership_authorized_by` fallback is architecturally coherent but is exercised at nearly every organization-affiliation and profile-media mutation site — a single point of trust, good for auditability, bad for blast radius if any one caller gets it wrong.
- **Dependency direction — clean.** A guardrail test confirms `apps.kernel.permissions` never imports from business apps. `apps.portal`'s one dependency on `apps.public_site` is one-directional and documented as such.
- **Extensibility — two structural dead ends worth naming.** `DiscoveryService.search()` presents itself as "the single public entry point" in its own docstring, but both directory services bypass it entirely (for a documented, defensible reason). Separately, `AllocationCalculator` and `SettlementBatch` are both real, tested primitives with no live consumer.
- **Scalability.** The one concrete, previously-undetected scalability risk: `SupplierSearchService.filter_suppliers()` has no database-level `LIMIT`. As the active-supplier count grows, every directory/search request materializes and ranks the full matching set in Python before slicing to a page of 12 — independent of, and not fixed by, the KL-012 query-count remediation.

## 7. Final status at assessment time

**Highest risks, ranked:**

| Severity | Risk | Why it matters |
|---|---|---|
| Critical | RBAC kill-switch has no admin UI or audit trail | The one flag that can disable all authorization tenant-wide is invisible to every operator surface in the repo |
| High | OTP has no SMS provider | Blocks every phone-based login/registration outside `DEBUG` |
| High | No real PSP anywhere | Blocks real money collection entirely; customers can currently self-mark their own payments succeeded |
| High | Escrow release has no wallet-crediting consumer | Latent today (feature gated off); would strand money the moment a tenant enables it |
| High | Commission allocation math is unwired | Contract/rate infrastructure fully built; nothing in the settlement path calls it |
| High | Order cancellation has no permission check | Any caller reaching `approve_cancellation()` can terminate any order |
| Medium | No tenant-scoping middleware | Isolation is currently sound but depends on every future query remembering `tenant_id` |
| Medium | Unbounded candidate-set materialization in discovery | A distinct, previously undetected scaling ceiling separate from the already-fixed query-count issue |
| Medium | CI never executed; no production deployment path | No automated proof any gate has passed in its real environment; no way to actually ship today |

**Recommended next implementation priority (as stated at assessment time):** the
existing roadmap's own next milestone — a Phase 5 Architecture Assessment for the
Marketplace Order Workflow / `OrderOfferService` — is corroborated as a genuine,
self-aware Not Started item, not a documentation gap. A pure risk-based reading
would sequence differently, starting with the RBAC kill-switch visibility gap
(smallest fix, highest blast-radius reduction), then OTP's SMS provider, then the
order-cancellation permission check, then a decision on the escrow/commission
wiring before either flag is ever enabled for a real tenant, then real CI
execution and a minimal deployment path — with the roadmap's own Phase 5
Architecture Assessment following after.
