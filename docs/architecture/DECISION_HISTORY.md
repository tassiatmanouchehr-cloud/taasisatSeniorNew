# Decision History

Status: current as of the Epic 02 — Marketplace Operational Experience
sprint (branch `claude/epic-02-marketplace-operational-experience`), based
on `main` @ `73bb852ceeff3c551476a628a283a56248abdb6d` (PR #23's merge
commit).

This is **not** another ADR. It is an index of every ADR in the
repository — where to find it, what it decided, why, and whether it still
holds. Read the linked ADR for the full Context/Decision/Consequences
reasoning; this table exists so a new developer can scan the whole
decision history in one page before diving into any single document.

Dates are the month the underlying decision was actually made (verified
against `git log`), which for several ADRs predates the ADR *document*
being written — several of these were formally recorded during the
architecture-consolidation sprint (Module 18) to document decisions
already made in earlier modules. Where that's true, it's noted.

| 2026-07 | Customer Experience Phase 1: reconcile `CareRecipient` with pre-existing `ElderProfile` by extending in place | — (no numbered ADR; decision recorded here and in `GAP_ANALYSIS.md`) | ADR-008 named a *future* `CareRecipient` entity without knowing `ElderProfile` (an older, pre-existing model already FK'd from `Order.elder_profile`) would still be present when that future arrived. Introducing a second, parallel model would have violated the sprint's own "no duplicate models" rule. Extending `ElderProfile` in place — new fields only, same class name — satisfies ADR-008's shape while reusing the existing FK and avoiding duplication. Product/UI vocabulary says "Care Recipient"; the Django model stays `ElderProfile`. | **Active** — `apps.accounts.models.profiles.ElderProfile`, `apps.accounts.services.care_recipients.CareRecipientService` |
| 2026-07 | Customer Experience Phase 1: extend the ADR-007 thin-controller guardrail to `apps.portal` | — (no numbered ADR; extends ADR-007) | A post-merge-review architecture pass found `apps/portal/views.py` calling the ORM directly in nine places (multi-row reads with no service in front of them) — exactly what ADR-007's `ApiViewOrmDisciplineTest`/`AdminPortalOrmDisciplineTest` exist to prevent, but no equivalent guardrail covered the new app. Fixed by adding `OrderQueryService`/`CatalogQueryService`/`OrderTimelineService` (`apps.orders`) and `NotificationQueryService` (`apps.notifications`), moving order-status timeline interpretation out of the portal layer into `apps.orders`, and adding `PortalOrmDisciplineTest`. | **Active** — enforced by `PortalOrmDisciplineTest` at test time, mirroring `AdminPortalOrmDisciplineTest` |
| 2026-07 | Epic 02: Provider Portal built supplier-generic, on `kernel.ServiceSupplier` — never `CaregiverProfile` directly | — (no numbered ADR; extends ADR-001's `ServiceSupplier` freeze) | `apps.provider_portal` must be reusable later for nurses, physiotherapists, technicians, and other supplier types without rewrite, per this epic's explicit constraint. `apps.accounts.services.provider_identity.resolve_supplier_for_user()` is the one, single call site that turns a `UserAccount` into a `ServiceSupplier`; the portal app itself imports no concrete profile model. | **Active** — `apps.provider_portal` has zero imports of `CaregiverProfile` (verified by grep); all identity resolution goes through `resolve_supplier_for_user()` |
| 2026-07 | Epic 02: provider assignment actions and organization assignment strategy are explicit, extensible service-level transition tables, not inline status writes | — (no numbered ADR; mirrors the pre-existing `apps.payments.services.transitions` pattern) | The epic required that future actions (counter-offer, delay request, escalation, replacement, and future assignment strategies like automatic/bulk/shift) be addable without breaking the existing contract. `apps.booking.services.provider_actions.ALLOWED_PROVIDER_TRANSITIONS` (one dict key per named action: `confirm`, `decline`) and `apps.booking.services.organization_assignment.OrganizationAssignmentService` (one named strategy, `assign_manual`, behind a service boundary designed to hold more) both follow this shape. Neither introduces a second assignment/booking engine — both call the unmodified `AssignmentService`/`ExecutionService`. | **Active** — `apps.booking.tests.test_provider_actions`, `apps.organization_portal.tests.test_assignment_center` |
| 2026-07 | Epic 02: `apps.provider_portal`/`apps.organization_portal` use ownership, not RBAC permission keys, as their security boundary | — (no numbered ADR; extends the `apps.portal` precedent) | The epic's suggested permission keys (`provider.assignments.act`, `organization.staff.read`, etc.) were not implemented literally as RBAC keys. `PermissionService.require()` calls would fail closed for any real actor without a seeded role assignment, and no infrastructure exists yet to seed org/provider-scoped `RoleAssignment.scope_type`/`scope_id` rows. Instead, `resolve_supplier_for_user()` (provider) and `resolve_organization()` (organization, via `list_administered_organizations(user).first()`) resolve "my own data" directly, mirroring `apps.portal.permissions`'s established pattern — a customer/provider/org-admin is always allowed to see their own data and never anyone else's. `OrganizationAssignmentService.assign_manual()` calls `AssignmentService.assign()` without `assigned_by` (system context, always allowed per existing `PermissionService` semantics) while separately publishing `OrganizationAssignmentChanged` with the real actor's `person_id` for traceability. | **Active** — a deliberate, documented trade-off, not an oversight; revisit once org/provider-scoped role-seeding infrastructure exists |
| 2026-07 | Epic 02: assignment-confirm → execution-session-creation orchestration lives in `apps.provider_portal`'s view layer, not inside `apps.booking` | — (no numbered ADR; respects the existing one-way dependency graph) | `apps.execution` already imports from `apps.booking` (`apps.execution.services.session_service` imports `SupplierAssignmentStatus`), so `apps.booking` must never import `apps.execution` back. Confirming a provider assignment and creating the corresponding `ExecutionSession` are two calls the portal view makes in sequence, not a single service-layer call that would require crossing that boundary. | **Active** — `apps.provider_portal.views.assignment_confirm_view`; see `dependency-graph.md` |
| 2026-07 | Epic 02: `CustomerRequestSubmitted` was not added as a new domain event | — (no numbered ADR) | The epic's candidate event list named `CustomerRequestSubmitted` alongside several others. `apps.orders.services.order_creation.create_public_order()` already publishes `ORDER_CREATED` unconditionally on every order creation, with a full audit trail via the existing `DomainEvent`/`publish()` pipeline. A second event describing the same state transition (a new order entering the system) would have been a duplicate, not an addition. | **Active** — `ORDER_CREATED` remains the canonical event for order/request submission |

## Reading order

New to this decision history? Read them in the table's date order — each
one either narrows or corrects something ADR-001 froze, and later ADRs
sometimes reference earlier ones (ADR-008 assumes the multi-role account
model that ADR-001's identity chain made possible; ADR-005 assumes the
`FinancialParty` abstraction ADR-004 relies on).

For what's actually built against each of these decisions today, see
[`PROJECT_MODULE_STATUS.md`](PROJECT_MODULE_STATUS.md). For what's still
